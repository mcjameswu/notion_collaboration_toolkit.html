01｜System Overview

這份 Toolkit 要解決什麼問題？

把你的身體條件、訓練背景、傷病限制、器材、課表格式、Notion 與 GitHub Pages 工作流，整理成一個可長期協作的 AI 健身系統。

INPUT

你的個人條件

年齡、身高、體重、體脂、訓練紀錄、教練課、器材與身體限制。

→
PROCESS

AI 課表決策

依今天時間、地點、疲勞與左肘／右腳踝／左大腿狀態，產出合適課表。

→
OUTPUT

HTML + Notion 紀錄

可瀏覽課表、可嵌入 Notion、可月度復盤並回饋下次課表。

一句話總結

Toolkit 負責記住你是誰；AI 負責根據當下狀態幫你做決策；Notion 負責保存紀錄；每月復盤負責讓系統越來越準。

可持續

用 Notion 持續更新身體資料、課表版本、訓練紀錄與月度回顧。

可追蹤

以訓練日誌追蹤重量、組數、次數、疲勞、恢復與傷病反應。

可視覺化

將課表與工具包輸出為 HTML，透過 GitHub Pages 發布並嵌入 Notion。

可傳遞

任何新 AI 工具讀取後，都能快速理解你的長期訓練脈絡。

03｜Safety Constraints

傷病與訓練注意事項

未來 AI 排課表時，必須先判斷左肘、右腳踝與左大腿外側狀態，再決定是否進階、維持或降階。

左手肘

約兩年前曾脫臼，已完整復健。一般訓練正常，但接近最大負荷時可能有輕微感覺。

  • 重推、重拉、負重支撐需監測。
  • 避免左肘不適時大量爆發伏地挺身。
  • 若尖銳痛或不穩，改用機械式或啞鈴變化。

右腳踝

去年 7 月曾嚴重扭傷，目前重訓基本可執行，但部分活動仍可能有些微異樣感。

  • 跳躍、單腳穩定、變向需保守漸進。
  • 若酸痛或不穩，取消高衝擊跳躍。
  • 訓練前加入踝關節活動與小腿啟動。

左大腿外側

近期較明顯，常在深蹲重量約 60kg 以上、蹲類每週 2–3 次以上或大量膝髖屈曲後出現。

  • 深蹲、腿推、弓箭步需控制容量。
  • 緊卡明顯時改髖主導或低膝屈曲版本。
  • 建立固定暖身與訓後恢復流程。
檢查項目綠燈黃燈紅燈
左肘無痛、推拉正常接近大重量有悶感尖銳痛、鎖住或明顯不穩
右腳踝無痛、單腳站穩、跳躍可控有酸或單腳動作不安定疼痛、腫脹或走路不適
左大腿外側無緊繃或輕微感覺深蹲時卡但可控制疼痛、卡住或影響軌跡
空腹與疲勞精神穩定、可集中能量偏低或睡眠不足頭暈、冒冷汗、心悸或動作失控
04｜Training Principles

訓練目標與課表設計原則

目前不是以短期減重或流汗為主，而是建立長期可執行、能提升最大肌力、爆發力與日常功能能力的訓練架構。

1. 最大肌力

最重要目標。以深蹲、臥推、硬舉、划船、肩推、輔助引體向上等複合動作為核心。

5×56×63RM / 5RM

2. 爆發力

第二目標。以壺鈴擺盪、高拉、上搏、抓舉、低衝擊爆發動作與短時間高品質輸出為主。

SwingHigh PullClean

3. 功能抗老化

長期目的。把訓練轉化到搬重物、爬樓梯、親子活動、旅行與身體自主能力。

核心平衡關節穩定
漸進超負荷規則

同組數同次數下，每 2–4 週嘗試小幅增加重量。若無法增加重量，就增加總次數、增加一組、縮短休息或改善動作速度。每 4–6 週安排減量週。

05｜Weekly Structure

目前週訓練結構與排程標準

預設使用 1-3-5 節奏，其中週三益身 GYM 教練課是固定核心訓練節點。

週期位置優先版本目標課表重點
第 1 次健身房自主訓練主力肌力日深蹲、硬舉、臥推、划船擇 1–2 個主軸，建立本週強度。
第 2 次週三教練課教練帶課主訓練依葉教練安排，課後記錄動作、重量、組數與身體反應。
第 3 次健身房優先，居家替代爆發力 + 補強 + 功能性根據週三內容補足未訓練模式，避免重複堆疊疲勞。

週三教練課後調整

  • 重胸推／肩推後：下一次避免大量胸肩推。
  • 重深蹲／腿推後：下一次避免大量膝主導下肢。
  • 重硬舉／後鏈後：下一次避免大量髖鉸鏈。
  • 大量背部拉後：下一次降低划船與單槓總量。
  • 全身高容量後：下一次採恢復日或神經激活日。

每次排課前必問

  1. 今天可訓練幾分鐘？
  2. 今天在健身房或居家？
  3. 上一次教練課做了什麼？
  4. 左肘、右腳踝、左大腿外側是綠燈、黃燈或紅燈？
  5. 今天是最大肌力、爆發力、恢復、技術或混合？
  6. 是否長時間空腹且能量偏低？
06｜HTML Standard

HTML 課表系統規範

HTML 課表的核心價值是：可分享、可閱讀、可互動、可嵌入 Notion，也能成為未來 AI 產生新課表的固定模板。

項目規格理由
檔案形式單一 .html 檔案，CSS 內嵌，JavaScript 非必要不用最容易上傳 GitHub Pages、備份、修改與傳給 AI。
語意結構使用 header、main、section、article、table、nav方便 AI、搜尋、轉 PDF 或改成簡報。
視覺風格清楚層級、卡片式區塊、色彩只用於重點支援口頭說明、內部 briefing 與手機閱讀。
列印友善加入 @media print方便轉 PDF、歸檔、列印。
行動裝置手機版改單欄，表格改卡片式堆疊訓練時多半用手機查看。

動作詳解欄位

  • 目的
  • 準備姿勢
  • 執行步驟
  • 正確感受
  • 常見錯誤
  • 替代動作

課表主體欄位

  • 階段與時間
  • 重量、組數、次數
  • 休息時間與 RPE
  • 高能量與低能量調整
  • 安全降階規則

設計頁面欄位

  • 課表主體
  • 設計邏輯
  • 漸進策略
  • 安全調整
  • 訓練後紀錄

月度復盤重點

  • 最大肌力是否進步。
  • 爆發力訓練是否足夠。
  • 左大腿外側是否與深蹲頻率、重量或膝屈曲量相關。
  • 左手肘是否受重推、重拉或撐體影響。
  • 右腳踝是否受跳躍或單腳訓練影響。
  • 蛋白質與恢復是否足夠。
  • 下個月 1-3-5 結構如何調整。
09｜AI Collaboration Rules

未來 AI 協作規則

任何 AI 協助產出課表前，都應先讀本 Toolkit，並用以下規則避免泛用、危險或無法執行的課表。

AI 應優先遵守

  • 最大肌力是第一目標。
  • 爆發力是第二目標。
  • 功能性與抗老化是長期目的。
  • 週三教練課是固定核心訓練節點。
  • 週一、週五必須配合週三內容調整。
  • 健身房版優先，居家版作為替代。
  • 不確定資訊要標示「待確認」。

AI 不應該做

  • 忽略左大腿外側緊繃問題。
  • 把不明確器材重量當成絕對負重。
  • 假設目前可以做完整自體重引體向上。
  • 在 30 分鐘內硬塞太多動作。
  • 把減脂排在最大肌力之前。
  • 右腳踝狀態不明時安排大量跳躍。
  • 左肘接近最大負荷不適時安排大量重推重拉。

請先閱讀 Fitness AI Toolkit，並根據以下資訊設計今天課表。

10｜Update Rules

何時更新 Toolkit

Toolkit 不是一次性文件。只要身體、器材、目標、課表格式或 Notion/GitHub 流程有變，就要更新。

類型更新時機處理方式
身體資料體重、體脂、血壓、心率有明顯變化。更新目前身體狀況與訓練風險判斷。
傷病狀態左肘、右腳踝、左大腿外側狀態改變。更新交通燈規則與替代動作。
訓練目標目標從肌力轉為減脂、耐力、體態或其他方向。重新排序訓練優先順序。
訓練頻率每週訓練天數、教練課或自主訓練安排改變。更新 1-3-5 架構與週期邏輯。
器材與場地新增器材、換健身房、彈力繩固定點確認。更新健身房版與居家版可用動作。
HTML / Notion / GitHub模板格式、發布流程、Notion 資料庫欄位改變。更新 v1.3 標準模板與工作流。
更新原則

新增資訊不要只附錄在文末；必須整合回正確章節，讓未來 AI 可以在最短時間內讀懂並直接使用。
